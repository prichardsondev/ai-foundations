Session 4 example

Codex App: A Real Agent Workspace

Codex is a coding-focused AI agent, but the important class idea is broader. It shows what agent-style work looks like when an AI system has a goal, tools, files, permissions, actions, and human review.

Use this page to notice

  • How an agent works inside a real project workspace
  • Why tools and permissions matter as much as the AI model
  • How review, tests, diffs, commits, and pull requests keep humans in charge
  • Why repeated agent tasks need narrow scope and safe boundaries

1. What Codex App is

A coding example that helps us understand agents more generally.

Codex is an AI agent for software work. In Codex App, a person can give Codex a task inside a project, and Codex can inspect files, reason through steps, use tools, suggest edits, run checks, and hand the result back for review.

For this course, the main lesson is not "AI can write code." The main lesson is that an agent works inside a bounded workspace. It needs the right files, the right tools, the right settings, and a person who decides what becomes final.

Access note

Codex requires signing in with a ChatGPT account that has Codex access. Use of Codex may count toward your plan's agentic usage limits.

2. Why it is an agent example

Use the Session 4 formula: agent = goal + tools + steps + actions + guardrails.

Goal

The user gives Codex a task, such as review this page, fix this issue, explain this file, or prepare a small change.

Tools

Depending on setup, Codex can inspect project files, use a terminal, use browser or app tools, and connect to approved outside systems.

Steps

Codex can plan, read, edit, test, revise, and explain what it changed. The work is more than one answer in one chat bubble.

Actions

Codex can propose file changes, run checks, help with Git tasks, prepare commits, or help create a pull request when the project allows it.

Guardrails

Approvals, sandbox settings, project scope, network access, tool access, and human review define the boundaries. More access can be useful, but it also means more responsibility.

3. Projects

A project is the workbench the agent can see.

In Codex App, a project is connected to a codebase or working folder. That project gives Codex the local context it needs: files, folders, commands, and project history.

A simple analogy: if Codex is the helper, the project is the workbench. The agent can only do useful work if the right materials are on the workbench and the boundaries are clear.

4. Chats and threads

Different threads can focus on different tasks.

Some Codex conversations are project-scoped, meaning they are tied to a specific project or worktree. Others can be useful for planning, research, triage, or discussing an idea before making changes.

Different threads can work on different tasks. One thread might review a lesson page, another might draft a small update, and another might investigate why a test fails. The thread keeps the task, messages, and work together.

5. Settings and permissions

Settings are agent boundaries, not just preferences.

Approval mode

Approval settings decide when Codex must stop and ask before it takes an action. This matters when work could change files, use network access, or affect another system.

Sandboxing

A sandbox limits what Codex can touch. For example, it may be able to work inside the current project but need approval to reach outside that workspace.

Project scope

Allowed directories and project scope define the files Codex can inspect or edit. A smaller scope is easier to understand and review.

Network and tools

Network access, browser tools, app connectors, notifications, and other tools should match the task. Do not give broad access just because it is available.

Safety message

More access means more responsibility. A strong agent setup gives Codex only the access it needs, keeps risky actions visible, and makes review easy.

6. Plugins, MCP, and skills

These extend what Codex can do.

Plugins are ways to package reusable workflows and app integrations for Codex. Official examples include plugins for Gmail, Google Drive, and Slack. Those kinds of connections can let an agent read information from outside tools or help take approved actions in them.

MCP stands for Model Context Protocol. In plain English, it is a standard way for AI tools to connect to other tools and systems.

A plugin can also include MCP servers, which give Codex access to additional tools or shared information outside the local project.

Skills are reusable instructions and resources for a specific kind of work. A skill can help Codex follow a known workflow reliably, such as using the right references or helper scripts for a task.

7. Automations

Automations let Codex run repeated tasks on a schedule.

Examples

  • Check a project each morning and summarize changes
  • Review recent changes and flag possible issues
  • Run a repeated documentation check
  • Watch a thread or project for a recurring task

Safe automation habits

  • Keep the task narrow
  • Use the smallest useful project scope
  • Use safe permissions by default
  • Review findings before accepting changes

Key idea

A scheduled agent is powerful because it can keep checking or improving something over time. Test the automation prompt manually first, then review the first few outputs before trusting a scheduled task.

8. Human review

The agent can do work, but the human decides what becomes real.

In software work, a diff shows exactly what changed. Tests check whether the project still behaves as expected. A commit records an approved set of changes. A pull request is a place where people can review and discuss changes before they are merged into the main project.

Those ideas apply beyond coding too. When an agent drafts work, connects tools, or prepares an action, a person should be able to inspect the result, understand the change, and accept, revise, or reject it.

9. Classroom demo

A simple way to demonstrate agent-style work without turning class into a coding tutorial.

  1. Ask Codex to review session-4.html without changing anything.
  2. Ask Codex to suggest one small improvement.
  3. Ask Codex to make the change.
  4. Review the diff.
  5. Accept, revise, or reject the change.

Prompt 1

Review session-4.html for teaching flow. Do not make changes yet. Summarize what the page teaches, where the flow is strong, and where students may need a clearer real-world agent example.

Prompt 2

Add a short student-facing section that links Codex App to the idea of an agent workspace. Keep the tone non-technical. Do not turn this into a coding tutorial.

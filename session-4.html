Session 4

GPT Chat Models vs AI Agents

This session explains the difference between asking AI for an answer and setting up AI to work through a task. The goal is to help you spot where chat is enough, where workflow automation helps, and where agent-style systems can add real business value.

Session outcomes

  • Explain the difference between a chat model, a workflow, and an agent
  • Recognize practical 2026 examples of agent-style work
  • Choose when automation is useful and when human review must stay involved
  • Identify safe first steps for non-technical users
  • Complete one simple in-class activity and inspect a real API in the browser

What this session covers

  • Chat model vs AI agent in plain English
  • Examples that show how agents are used in real work
  • Workflow automation vs agent-style systems
  • Where caution and human approval matter most
  • How beginners can get started without building a giant system
  • A hands-on task classification activity

1. The basic difference

Both use AI, but they are not the same kind of tool.

Chat model

A chat model is mainly there to answer, draft, explain, rewrite, summarize, and help you think.

You ask for something. It responds. In most cases, you stay in charge of the next step.

AI agent

An agent is closer to a system working toward a goal.

It may have tools, files, instructions, steps, and rules. It can gather information, make decisions inside limits, take actions, and hand work back to a person for approval.

2. A simple way to remember it

Use this mental model when someone says “agent.”

  • Chat model = answers, drafts, explains
  • Agent = goal + tools + steps + actions + guardrails
  • Workflow automation = fixed path with known steps
  • Many real systems combine all three

3. A few tech terms you should know

You do not need to be a programmer to understand agents, but these terms will help when you use AI tools or workflow platforms such as n8n.

API

An API is a way for one program to talk to another program.

Example: an AI system might use an API to read a calendar, check a database, send a message, or call an AI model.

HTTPS

HTTPS is the secure version of a website connection.

It helps protect data as it moves between systems, which matters when agents handle business information.

Authentication

Authentication means proving who you are.

Examples include signing in, using a token, or using an API key.

Permissions

Permissions determine what a user or system is allowed to access or do.

An agent may be allowed to read a file but not delete it. Good permissions reduce risk.

JSON

JSON is a common text format used to send structured data between systems.

If you use APIs, automation tools, or AI integrations, you will probably see JSON often.

Webhook

A webhook is a way for one system to notify another when something happens.

Example: when a new form is submitted, a webhook can trigger a workflow automatically.

Workflow automation

A workflow is a series of steps a system follows.

Example: receive an email, classify it, save it, and notify a person. This is often different from an agent, which may choose among tools and adapt its steps.

Quick comparison: API vs webhook vs workflow

API

Simple meaning: A way for one system to ask another system for data or an action.

Easy example: An app asks the basement temperature API for the latest reading.

Webhook

Simple meaning: A message sent automatically when an event happens.

Easy example: A form submission triggers a workflow the moment the form is completed.

Workflow

Simple meaning: A set of connected steps that move work from one stage to the next.

Easy example: A new request comes in, gets categorized, logged, and assigned to a person.

Why these terms matter

Agents usually depend on these building blocks. They connect to tools, work through secure web systems, use APIs, follow permissions, and take action inside workflows.

4. Real Example: Sensors, APIs, and Automation

This classroom demo uses real hardware, real data, and a real public API.

The basement demo in plain English

We built a small basement temperature demo using two Raspberry Pis and DHT22 sensors.

One Raspberry Pi is on the far side of the basement. The other is near the pellet stove. The public-facing Pi is exposed through Cloudflare at basetemp.raspberrynode.com.

That public API can return one local reading, both sensor readings together, or a simple decision about whether the stove is likely running.

Try these endpoints in a browser

  • https://basetemp.raspberrynode.com/api/temp = one raw local sensor reading from the far-side Pi
  • https://basetemp.raspberrynode.com/api/all = both raw sensor readings returned as JSON
  • https://basetemp.raspberrynode.com/api/status = fixed coded logic that returns LIKELY RUNNING or LIKELY OFF

Example JSON

{
  "far_side": {
    "temperature_f": 66.2,
    "humidity": 42.1
  },
  "near_stove": {
    "temperature_f": 78.9,
    "humidity": 31.4
  },
  "time": "2026-04-06T14:05:00Z"
}

A simplified /api/all response like this shows both raw readings in JSON. This is the kind of data a browser, app, workflow, or AI tool can read from an API.

Where APIs show up

  • Websites and cloud services
  • n8n and other workflow tools
  • Calendars, email, Slack, Teams, and CRMs
  • Databases, sensors, and AI systems

Why /api/status is automation, not yet a full agent

It follows the same fixed process every time:

  • Read sensor A
  • Read sensor B
  • Compare the values
  • Apply a threshold
  • Return a result

This uses APIs, but the /api/status result is still coded automation, not yet a full agent.

That is useful automation, but it is still fixed logic rather than a system adapting its own approach.

What an AI or agent version could add

  • Look at the last hour or day of readings
  • Detect unusual patterns instead of one simple difference
  • Generate a plain-English explanation with uncertainty
  • Decide whether a person should be alerted
  • Call another system to log, message, or report the result

Quick browser activity

Open /api/temp, /api/all, and /api/status in a browser. Then identify which one is raw data, which one combines both systems, and which one applies logic before returning a result.

5. What this looks like at work

Now that you know the basic building blocks, here is what agent-style work looks like in normal business situations.

The best examples are not science fiction. They are ordinary business tasks.

Chat-based examples

  • Draft a reply to a difficult email
  • Summarize a meeting transcript
  • Explain a spreadsheet formula in plain English
  • Rewrite a policy so it is easier to understand
  • Brainstorm questions for a customer interview

Agent-style examples

  • Process incoming email, sort it by type, draft responses, and route the rest to the right person
  • Prepare for calendar meetings by reading notes, reviewing related files, and creating a short briefing
  • Read folders of documents and produce structured summaries or action lists
  • Analyze spreadsheets, spot exceptions, and draft a manager update
  • Monitor Slack or Teams channels for issues, summarize patterns, and assign follow-up tasks

6. Why agents get attention

Agents matter when the work is bigger than one answer.

Plain-English example

A chat tool can help write one email. An agent-style system might read the email, check related documents, look at the calendar, draft a response, update a task list, and ask a human for approval before sending anything.

The key idea is not “autonomy for everything.” The key idea is coordinated help across multiple steps.

That is why agents are often discussed alongside tools, permissions, approvals, and monitoring.

7. Workflow automation vs agents

Do not jump to agents if a simpler workflow will do the job.

Workflow automation may be enough when...

  • Steps are known in advance: The path is fixed and repeatable.
  • Tool choice is already set: The sequence is mostly decided ahead of time.
  • Reasoning is limited: The system mainly follows rules.
  • Exceptions are rare: The job is predictable.
  • Good starting advice: Start with a workflow tool such as n8n when the process is mostly known and fixed.

An agent may be more appropriate when...

  • Steps are not fully known: The system must figure out which path fits.
  • It must choose among tools: The best next step depends on the situation.
  • It needs multi-step reasoning: It must gather, compare, decide, and adapt.
  • Exceptions happen often: The task changes based on what it finds.
  • Good starting advice: Consider an agent when the system must reason through steps, choose tools, and adapt.

Important reality

Many real business solutions combine both approaches. A workflow might handle the predictable parts, while an agent handles the parts that require judgment, reading, classification, or adapting to new information.

8. Automation Tools and Agent Tools

Different tools fit different kinds of work.

Automation / workflow tools

  • n8n: A flexible workflow tool for connecting apps, APIs, and logic when the steps are mostly known.
  • Make: A visual automation platform that helps you link services and build step-by-step processes.
  • Zapier: A beginner-friendly automation tool for connecting common business apps and repetitive tasks.
  • Apache Airflow: A more technical workflow scheduler often used for data pipelines and repeated backend jobs.

Agent-style tools

  • ChatGPT agent: Can work through a task with tools, files, and multiple steps inside clear limits.
  • Codex app: An agent-style coding tool that can inspect a codebase, make changes, and work toward a goal.
  • Claude Code: A coding-focused agent that can reason through software tasks and use tools while you review the work.
  • Antigravity (optional): An agent-first coding environment built around goal-driven work rather than single prompts.

Simple rule of thumb

Workflow tools are best when the steps are mostly known. Agent tools are best when the system needs to choose tools, adapt, or work through multiple steps. Many real systems combine both.

9. Where agents are heading in 2026

The direction is practical, not magical.

2026 direction

The strongest trend is toward narrow, useful, trustworthy agents. Businesses are looking for systems that save time on a specific job, work inside clear limits, and make human review easy.

The direction is less about flashy autonomy and more about dependable systems that fit real work.

  • More agents will be built around one role or one workflow, not “do everything” promises
  • Trust, permissions, approval steps, and audit trails will matter as much as raw intelligence
  • Packaged business tools will continue to make agents easier for non-technical teams to try
  • Reliable handoff to a human will remain part of good system design

10. When agents are useful in real business settings

Use agents where the value comes from combining information and helping move work forward.

Good use cases

  • Preparing meeting briefs from calendars, notes, and documents
  • Sorting, summarizing, and routing incoming requests
  • Reviewing files or spreadsheets for patterns, exceptions, or missing details
  • Monitoring communication channels and surfacing items that need attention
  • Creating first drafts of summaries, updates, and task lists

What makes these work well

  • The task is narrow enough to define clearly
  • The system has the right inputs and tool access
  • There are clear rules for what it can and cannot do
  • A person can review important actions before they happen
  • Success can be measured in time saved, accuracy, or consistency

11. Where human review matters most

This is where good judgment beats speed.

  • Sending email to customers, vendors, or leadership
  • Modifying records in a system of record
  • Purchases, refunds, or anything involving money movement
  • Scheduling or changing calendar events that affect other people
  • External communication of any kind
  • Legal, financial, medical, compliance-related, or high-trust decisions

Good rule

If the action could create cost, risk, confusion, or harm, add human approval before the final step.

12. How non-technical users can get started

Start with packaged tools, not a giant custom build.

Your best first project is usually one repeated task, not a full business process.

Good first steps

  • Try packaged tools first, such as ChatGPT agent features or Microsoft Copilot agents and Agent Builder
  • Start with one narrow task that happens often
  • Choose a task where success is easy to judge
  • Keep a person in the loop for anything risky
  • Improve the process before trying to expand it

A smart beginner approach

  • Pick one job such as meeting prep, inbox triage, or weekly summaries
  • Define the inputs, outputs, and approval point
  • Test with a small number of examples
  • Watch for errors, unclear instructions, and missing context
  • Only expand after the narrow version is dependable

13. Hands-on activity

This activity helps you decide what kind of AI support fits a task.

Activity goal

Classify tasks as one of five options: chat model, workflow automation, agent, agent with human approval, or do not automate.

Part 1: Classify the task

For each example below, decide which category fits best.

  • Rewrite a rough email so it sounds professional
  • Move form submissions into a spreadsheet and notify a manager
  • Read a shared folder, summarize the files, and prepare talking points for a meeting
  • Watch a team channel for urgent issues and draft follow-up tasks for approval
  • Automatically send refunds without any human check

Part 2: Design a simple solution

Use this worksheet structure to sketch a simple solution.

  • Task: What job are we trying to complete?
  • Inputs: What information does the system need?
  • Tools: What systems or files would it use?
  • Output: What should it produce?
  • Approval point: Where should a human review the result?
  • Do not automate: What should stay fully human?

14. Discussion prompt

Bring the session back to real life.

Ask yourself: Where in your own work or life do you have a repeating task that involves reading information, making a simple decision, and preparing a draft or next step?

Then ask yourself: Should that task use a chat model, a workflow, an agent, or an agent with human approval?

15. Key takeaway

Keep the idea simple and useful.

  • Use chat when you mainly need answers, drafts, or explanations
  • Use workflow automation when the process is clear and repeatable
  • Use an agent when the system must work through steps, use tools, and adapt
  • Keep humans involved when the outcome affects money, records, trust, or compliance
  • Start small, stay practical, and build trust before expanding
